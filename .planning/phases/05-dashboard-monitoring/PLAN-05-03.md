### Task 6: Wire WebSocket for Execution Progress
**File**: `apps/frontend/src/hooks/use-websocket.ts`
- Add `useExecutionProgress(executionId: string)` hook:
  ```typescript
  export function useExecutionProgress(executionId: string) {
    const queryClient = useQueryClient();
    
    useWebSocket({
      events: {
        [`execution:${executionId}:step`]: (stepData) => {
          // Update execution in cache with new step status
          queryClient.setQueryData(['execution', executionId], (old) => ({
            ...old,
            steps: old.steps.map(s => 
              s.id === stepData.stepId ? { ...s, ...stepData } : s
            ),
          }));
        },
        [`execution:${executionId}:completed`]: (result) => {
          queryClient.invalidateQueries({ queryKey: ['execution', executionId] });
          queryClient.invalidateQueries({ queryKey: ['executions'] });
        },
      },
    });
  }
  ```
- Optimistic cache updates for step progress (no refetch needed per step)
- On execution completion, invalidate queries for fresh data

## Test Strategy
1. **Filter Combinations**: Test all filter permutations on executions page — single filter, combined filters, clear filters
2. **Search Behavior**: Type in workflow search, verify debounced API calls, results update
3. **Execution Detail**: Navigate to a completed execution, verify all steps shown with correct statuses
4. **Real-Time Progress**: Start a multi-step workflow, open execution detail, observe steps updating live
5. **WebSocket Disconnect**: Disable network briefly, verify graceful reconnection and state recovery
6. **Workflow Stats**: Compare card stats with actual database counts
7. **Empty States**: Test with no executions matching filters, verify user-friendly messages
8. **Pagination**: If >10 executions, verify pagination controls work
9. **Mobile Responsiveness**: Test filter bar stacking and card layout on narrow viewports
